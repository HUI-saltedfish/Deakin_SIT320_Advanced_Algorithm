Learning Goals

The module is designed to achieve following learning goals:

  • Understanding good software design principles.
  • Understanding the difference between good and bad design.

We have been working on algorithms so far in this unit. Our focus has been solely on the strategy, e.g., divide and conquer vs. dynamic programming vs. greedy etc. In this module (and the next), we will change focus. We would like to study what is a good written piece of code and what is not. And secondly, we would like to see if we can follow some guidelines or principles that can inform our decision during our software design process.

Software Development Cycle

In any typical product development, we have the phases of 'Analyse', 'Design' and 'Implement'. However, we know that software development is more of an agile and flexible process. And the three phases some what looses their distinction. For example, we can end-up making design decisions in the analysis phase, and revisit analysis in design. We can re-visit some of the design decisions in the implementation and so on. Unified Modelling Language (UML) is a good tool for structuring our analysis and design phases. If you are unfamiliar with UML, you should look at the optional section of this module on Unified Modelling Language.

Object Oriented Paradigm

We all must agree that once it comes to software development, there is one paradigm that has passed the test of time. That is object oriented paradigm. You already have done object oriented programming in SIT232 and SIT221, and know what does it entail. Basically, the idea is to think everything in terms of classes and objects. Each section of your code is an object that interacts with another section of your code which is also another object. Your program is nothing but a collection of various objects that interact with each other at the running time. 

In this module, when discussing 'Software Design' Principles', we will implicitly assume that we are following the object oriented paradigm. 

lightbox

Object-oriented development in the nutshell is the development of use-case diagrams in the Analysis phase, and the development of class-diagrams in the Design phase, whereas in the implementation phase, we implement the class diagram using the language of choice, say C++, Java, C#, Python, etc.

Design Principles

Since our analysis and design impacts our implementation - we must make sure that we make good decisions during analysis and design phase. Well, can we get some guidance, e.g., what is a good design and what is a bad design? One can answer these questions based on some design principles. If we follow these principles there is some guarantee that our design is fundamentally sound. So what are these principles? Well, in this module, we will look at:

  • Open-Closed Principle (OCP)
  • Dependency Inversion Principle (DIP)
  • Liskov Substitution Principle (LSP)

These are not the exhaustive list of principles that you can follow. But these are definitely the important one, one you should always adhere to when implementing any software program. 

Some other principles that you might be interested as part of your further reading and research includes:

  • Interface Segregation Principle
  • Single Responsibility Principle
  • Common Closure Principle
  • Common Reuse Principle
  • Stable Dependencies Principle
  • Stable Abstractions Principle
  • Reuse/Release Equivalency Principle
